# 需求澄清 Skill

## 目标

澄清用户需求并收集可验证的 Contract Case。这是**流水线中唯一与用户交互的阶段** — 后续所有阶段（RFC、拆解、执行）完全自动化，依赖此阶段产出的信息。

## 基座

调用 `superpowers:brainstorming` 进行意图探索和需求收集。遵循其方法论理解用户真正需要什么，但将退出条件从"进入 writing-plans"改为"产出 Contract"。

## 触发条件

- 用户在项目群发送需求消息
- 看板上创建新 Mission

## 输入

- 用户的原始需求消息
- 项目上下文（project.cwd, project.repoUrl）

## 执行流程

### 阶段一：分析与批量提问

1. 仔细阅读用户的原始消息
2. 一次性识别所有信息缺口 — 不要逐个提问
3. 通过 AskUserQuestion 按主题分批提问（每批一个主题领域，不是每轮一个问题）：
   - 这解决什么问题？谁受影响？
   - 范围边界：什么在范围内 vs 范围外
   - 现有行为要改变 vs 全新功能
   - 非功能需求（性能、兼容性、错误处理）
   - 用户关心的边界场景或约束

为什么要批量提问：逐个提问浪费用户时间。将相关问题分批（比如范围问题一批、合约问题一批），每批覆盖一个连贯主题，让用户可以一起思考相关内容。

### 阶段二：确认理解

4. 将精炼后的需求用一段简洁的文字总结
5. 如果仍有关键信息缺失（少见 — 阶段一应该覆盖大部分），再通过 AskUserQuestion 补一批
6. 批次数量根据需求复杂度弹性决定 — 简单需求 1-2 轮，复杂功能可到 4-5 轮。如果进展停滞，总结已有信息并明确标注剩余缺口

### 阶段三：Contract Case 收集

7. 简要解释 Contract Case 概念：
   > "为了验证结果符合你的预期，我们来定义几个具体的测试场景 — 每个包含特定输入和期望输出。"
8. 根据需求建议 1-3 个示例 Case（不要让用户从零发明）
9. 通过 AskUserQuestion 请用户确认、修改或补充 Case
10. 验证每个 Case 是具体且可验证的 — 拒绝模糊的"应该好用"类描述

### 阶段四：生成产出

11. 生成 `description.md` 和 `contract.json`
12. 写入 `{project.cwd}/.missions/{missionId}/`
13. 将最终需求摘要和 Contract 发送到飞书话题供用户确认

## 产出格式

### description.md
```markdown
# {Mission 标题}

## 问题
{解决什么问题，谁受影响}

## 范围
- **包含**: {范围内的内容}
- **不包含**: {范围外的内容}

## 功能需求
1. {需求项}
2. ...

## 非功能需求
{性能、兼容性、错误处理等约束 — 没有则省略}
```

### contract.json
```json
{
  "cases": [
    {
      "id": "case-1",
      "description": "...",
      "input": "...",
      "expectedOutput": "...",
      "type": "unit | integration | e2e"
    }
  ],
  "acceptanceCriteria": [
    "所有 Contract Case 通过",
    "现有测试不被破坏",
    "lint 和 build 通过"
  ]
}
```

## 约束

- 按主题分批提问 — 永远不要每轮只问一个问题
- 批次轮数根据复杂度弹性调整（简单 1-2 轮，复杂可到 4-5 轮）
- 必须至少产出 1 个 Contract Case
- 永远不要假设用户未确认的需求
- 如果用户请求很简单（错别字修复、配置变更），跳过阶段三，创建最小 Contract

## 退出条件

Mission 状态 → `inbox`，等待 Owner 审批。

Logs 模块全面升级 — 3 种方案

点击选择你偏好的方案，或在终端告诉我

A

方案 A — 稳健升级（推荐）

监控区：4 个 StatCard（总条目、错误数/率、模块数、最近错误时间）+ 按小时分布的迷你柱状图（纯 CSS，无依赖）

日志表：行内 accordion 展开详情 + 全文搜索 + 复制按钮 + Level 筛选提示（≥ 语义）

后端：新增 /api/v1/logs/stats 聚合接口，前端一次请求拿到统计

实时：手动刷新 + 可选自动轮询（30s 间隔 toggle）

优势

  • 零新依赖，纯 CSS 柱状图
  • 改动范围可控：1 个后端接口 + 前端重构
  • 对标 Traces 页的设计语言

限制

  • 柱状图相比 chart 库视觉丰富度有限
  • 轮询而非实时推送
B

方案 B — 专业日志查看器

监控区：4 StatCard + Recharts 折线图（按小时日志量趋势 + 错误率曲线）

日志表：虚拟滚动（react-window）+ 行内展开 + JSON 语法高亮 + 全文搜索 + 键盘导航

后端：stats 接口 + 流式 SSE 实时推送

实时：SSE 实时尾随（tail -f 体验）

优势

  • 最接近专业工具（Grafana Loki / Datadog Logs）体验
  • 大量日志下性能好（虚拟滚动）
  • 实时推送，延迟最低

限制

  • 新增 2-3 个依赖（recharts, react-window）
  • SSE 后端改动较大
  • 工作量约为方案 A 的 2-3 倍
C

方案 C — 渐进式（先 A 后 B）

第一阶段：实现方案 A 全部内容，立即可用

第二阶段：在 A 基础上升级：加 Recharts 图表、虚拟滚动、SSE 实时

架构：第一阶段就预留接口，第二阶段无需重构

优势

  • 快速出可用版本
  • 第二阶段可根据实际需求决定是否做
  • 风险最低

限制

  • 总工作量略多于一步到位
  • 需要两轮迭代

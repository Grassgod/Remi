Traces Module - Issues Analysis

Code review findings from dashboard-redesign branch

BUG 1 Total Traces = limit, not real count

Stats card shows traces.length which is capped at API limit (default 50). If today has 162 traces, card still shows 50.

Root: handlers/traces.ts returns LIMIT ? without a separate COUNT query
Total Traces
50
should be 162
BUG 2 Stats computed on truncated batch

Avg Duration, P95 Duration, Error Rate all calculated from the loaded 50 rows, not the full day's data. Stats are misleading when there are more rows than the limit.

Root: Traces.tsx computes stats client-side from traces array (capped at limit)
UX 3 "UNSET" status confusing for in-flight requests

When status = "processing", it maps to "UNSET" (OpenTelemetry term). Users don't know what "UNSET" means. Should show "Processing" or a spinner.

UNSET → Processing
UX 4 endTime fallback causes durationMs=0

When cli_round_end is null (processing traces), rowToTraceData falls back to created_at. This makes duration show "0ms" which drags down Avg/P95 calculations.

Root: tracing.ts:rowToTraceData() — endTime defaults to created_at when cli_round_end missing
MINOR 5 Waterfall labels overflow at 160px

Operation names like handle: [feishu] abc123def456 exceed the 160px SVG text area. No truncation or ellipsis applied.

handle: [feishu] abc...
overflows →→→→→
Summary
2 bugs (stats accuracy), 2 UX issues (status label + duration fallback), 1 minor (SVG overflow).
Bugs 1 & 2 are related — both stem from missing server-side aggregation (no COUNT/AVG/P95 SQL queries).

Skills + Wiki + Memory: 统一编辑体验

三个方案，核心区别在于代码复用层级

A

抽取 MarkdownFileViewer 共享组件 (推荐)

从 Memory 现有代码中提取一个通用的 <MarkdownFileViewer /> 组件，封装 Edit/Preview 切换 + Save + textarea/ReactMarkdown 逻辑。Skills、Wiki、Memory 三个页面都使用这个组件。

Pros

  • UI 一致性有保障，改一处全局生效
  • 各页面保留自己的数据获取和树结构逻辑
  • 改动量适中，不需要重构整个架构

Cons

  • 需要给 Wiki 新增写入 API
  • 组件 props 需要兼容三个页面的差异（面包屑、metadata 等）
B

统一文件浏览器模式

构建一个通用的 <FileBrowser />，包含 TreeSidebar + FileViewer + Editor 整套。Skills/Wiki/Memory 只负责提供数据源配置，UI 完全统一。

Pros

  • 最高度统一，三个页面几乎零重复代码
  • 未来新增类似页面几乎零成本

Cons

  • 过度抽象——Memory 有搜索/实体元数据、Wiki 有 git 历史、Skills 有 cron 状态，差异很大
  • 改动量大，需重构三个已有页面
  • 违反 YAGNI 原则
C

Skills 独立实现 + Wiki 最小改动

Skills 页面参照 Memory 的模式独立实现编辑功能。Wiki 页面只加一个 Edit/Save 按钮到现有 Card 里。不提取共享组件。

Pros

  • 最快交付，每个页面独立可测
  • 不动 Memory 现有代码，风险最低

Cons

  • Edit/Preview 逻辑重复三遍
  • 后续样式调整需改多处
